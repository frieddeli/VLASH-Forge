    5-Minute Presentation Script

    Distributed VLA Fine-Tuning on the Cloud: Dockerizing VLASH with DeepSpeed

    COMP4651 — Cloud Computing and Big Data Systems

    Slide 1 — Title (0:00–0:20)

    "Good [morning/afternoon]. Our project is called VLASH Forge.

    We built a portable, containerised distributed training platform for fine-tuning large robot learning models on cloud and HPC infrastructure — deployable to Kubernetes, AWS, GCP, and HPC clusters from a single Docker image and one command."

    Slide 2 — The Cloud Computing Problem (0:20–1:00)

    "The core cloud computing problem we're solving is this: fine-tuning a modern Vision-Language-Action model — a 1.3 billion parameter transformer that drives a robot arm — requires cloud-scale compute. We're talking 12 to 40 gigabytes of GPU memory per device, hours of training time, and a software stack that historically has never been publicly released.

    The VLASH framework is one of the best VLA models available. It was validated on 4 times H100 GPUs but its distributed training infrastructure was never open sourced. There was no reproducible, portable way to run it on cloud or HPC.

    Our contribution: we built that missing infrastructure."

    Slide 3 — What We Built (1:00–1:35)

    "Specifically, we implemented and packaged three things:

    One — DeepSpeed ZeRO-2 distributed training via HuggingFace Accelerate, replacing the unpublished DDP setup from the original paper.

    Two — A Docker and Singularity container with a pixi lockfile, guaranteeing bit-identical environments across cloud VMs, HPC nodes, and Kubernetes pods.

    Three — A Kubernetes Job manifest, making the full training pipeline submittable with a single kubectl apply on EKS, GKE, or AKS.

    Everything runs with one command — scripts/train.sh — which detects the environment and launches automatically."

    Slide 4 — System Architecture (1:35–2:20)

    "Here's the architecture.

    On the left, the infrastructure deployment. The container is pulled from Docker Hub. A job scheduler — PBS on HPC, SLURM on other clusters, or Docker Compose on a cloud VM — allocates a GPU node and starts the container. HuggingFace Hub provides the base model and training dataset at run start, and receives the final checkpoint when training completes. Weights and Biases streams live metrics throughout.

    On the right, the training pipeline inside the container. The base model and dataset feed into distributed training, which uses either DeepSpeed ZeRO-2 for LoRA fine-tuning or FSDP for full fine-tuning, and writes a checkpoint.

    Critically: nothing is baked into the image. All credentials, dataset IDs, and training parameters are injected at runtime via environment variables."

    Slide 5 — DeepSpeed ZeRO-2: The Core Contribution (2:20–3:00)

    "The core distributed training contribution is DeepSpeed ZeRO-2.

    The problem with vanilla DDP: every GPU holds the full model, all gradients, and all optimizer states. For a 1.3B parameter model, that's over 15 gigabytes per GPU before activations — you need 80GB cards just to run.

    ZeRO-2 shards the optimizer states and gradients across GPUs while keeping parameters replicated. On 4 GPUs, this drops the per-GPU footprint from 15.6 to 5.85 gigabytes — a 62% reduction. That's what lets us run batch size 16 on standard 40GB A100s.

    For full fine-tuning where even parameters need sharding, we support FSDP — PyTorch's equivalent of ZeRO-3 — selected by a single environment variable."

    Slide 6 — Kubernetes Deployment (3:00–3:25)

    "A key deliverable of this project is the Kubernetes Job manifest.

    The manifest defines GPU resource requests, PersistentVolumeClaims for model cache and training outputs, and a Kubernetes Secret for the HuggingFace token. The full training job is submittable with a single kubectl apply.

    PersistentVolumeClaims abstract the underlying storage — EBS on AWS, Filestore on GCP, Azure Disk on AKS — so the same manifest works across all major cloud providers without any changes."

    Slide 7 — Storage Abstraction (3:25–3:45)

    "The storage design is what makes the system portable across HPC and cloud.

    Every environment exposes storage differently — HPC uses Lustre or GPFS, cloud VMs use EBS or Persistent Disk, and Kubernetes uses PVCs. Rather than handling each case separately, our container always reads and writes to /scratch.

    The host bind-mounts whatever storage it has to /scratch. One environment variable — SCRATCH — is all that changes between an NSCC run and an AWS run. This composability extends to object storage: GCS or S3 buckets mounted via FUSE appear as regular directories to the container."

    Slide 8 — Results and Demo (3:45–4:40)

    "We validated the full pipeline end-to-end.

    Training was conducted on NSCC ASPIRE 2A — Singapore's national HPC facility — under the academic allocation programme, since no commercial cloud budget was available for the project. NSCC operates equivalently to managed cloud compute, providing on-demand GPU node allocation via a PBS scheduler.

    LoRA fine-tuning of pi-zero-point-five on 50 demonstrations took approximately 4 hours on 4 A100s, fitting within 12 gigabytes of VRAM per GPU — confirming that ZeRO-2 enables efficient multi-GPU training at batch size 16.

    The resulting checkpoint was deployed to a Piper arm running on a Jetson AGX Orin. Here's the deployment in action.

    [play demo] The arm completes pick-and-place continuously at 30Hz with a 65% task success rate — versus 5% with a synchronously trained baseline. Asynchronous inference reduced latency by 29.5 times."

    Slide 9 — Conclusion (4:40–5:00)

    "To summarise our cloud computing contributions:

    We implemented DeepSpeed ZeRO-2 distributed training for VLASH — the open infrastructure that was missing from the original paper.

    We packaged it in a Docker and Singularity container deployable to Kubernetes, NSCC ASPIRE, AWS, and GCP — with a unified storage abstraction that requires only one environment variable to change between environments.

    The code, Docker image, and documentation are all public. Thank you."
